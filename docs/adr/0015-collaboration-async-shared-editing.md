# 협업은 비동기 공유접근 + 비관적 편집 락으로 하고 실시간 동시편집은 보류한다

Project를 소유자(Owner)가 다른 User에게 **역할(편집자/뷰어)과 함께 공유**할 수 있게 한다. 단, 여러 명이 같은 ERD를 동시에 라이브로 편집하는 실시간 협업(CRDT/웹소켓/presence)은 v1 범위가 아니다. 대신 **비동기 공유접근**: 한 번에 **한 명만** 편집하도록 프로젝트 단위 **편집 락(lease)**으로 강제하고, 나머지는 읽기 전용으로 합류한다. 락의 좁은 경합 구간은 `project.version` **낙관적 백스톱**으로 막는다.

이는 `CONTEXT.md`가 명시했던 *"공유·협업은 범위 밖"* 전제를 바꾸는 결정이라 기록한다.

## Considered Options

### 편집 동시성 모델

- **비동기 공유접근 + 비관적 락 (채택)**: 편집자(소유자 포함)가 에디터를 열면 프로젝트 단위 편집 락을 잡고, 타인은 읽기 전용으로 들어간다. lost-update를 *애초에 두 편집 세션이 공존하지 못하게* 막는다. 기존 아키텍처(DBML 텍스트 단일 진실 ADR-0001 + 600ms 디바운스 autosave + 스냅샷 ADR-0014)에 그대로 얹힌다. 사용자 의도가 "초대해 함께 편집/읽기전용"이라는 *접근 권한* 요구였고 실시간 동기화 요구가 아니었다.
- **실시간 동시편집 (보류)**: 여러 명이 동시에 보며 변경이 즉시 전파. DBML 텍스트가 단일 진실인 구조에서 텍스트 CRDT/OT + presence + 별도 실시간 서버가 필요 — 규모와 위험이 별개 프로젝트급. 필요해지면 독립적으로 도입(이 결정을 마이그레이션 기점으로 삼는다).
- **낙관적 버전 체크 단독 (기각)**: 모두 편집 가능하되 stale 저장을 409로 거부. 가볍지만 "동시에 둘이 편집하다 한쪽이 거부당하는" 오류 경험이 사용자에게 노출 — "편의를 제한해 오류를 원천 차단"이라는 요구에 어긋난다. 단, 락의 보조 백스톱으로는 채택(아래).

### 락 만료·갱신·이어받기

- **가시성 기반 heartbeat + TTL + 소유자 강제 이어받기 (채택)**: 보유 중 탭이 보일 때만 ~20초마다 갱신, 무활동/숨김 시 TTL(60초)로 자동 만료 → 타인이 이어받기(takeover). 살아있는 락은 원칙적으로 강제 탈취 불가지만 **소유자만** 예외로 강제 회수 가능. 켜둔 채 방치한 탭이 락을 영원히 쥐는 문제를 가시성 게이팅으로 방지하고, 생각하느라 잠시 멈춘 편집자를 활동기반보다 덜 밀어낸다.
- **활동 기반 만료 (기각)**: 편집이 멈추면 만료 — 자리 양보엔 적극적이나 생각 중인 편집자를 밀어낼 위험.
- **version 낙관적 백스톱 병행 (채택)**: 모든 콘텐츠 쓰기 PATCH에 본 `version`을 동봉, 서버가 최신과 다르면 409. 락 만료↔이어받기 사이 좁은 경합 틈으로 들어온 stale 저장을 막는 2차 방어.

### 소유자·멤버 저장 구조

- **`project.user_id` 권위 유지 + `project_member`에 비소유자만 (채택)**: 소유자는 기존 단일 소유 컬럼 그대로, 멤버 테이블엔 편집자/뷰어만. 마이그레이션이 *테이블 추가만*으로 끝나고(기존 프로젝트 백필 불필요), 전 코드가 `user_id`에 의존하던 것을 깨지 않는다. 접근 판정은 `resolve_role(project,user)`로 통합.
- **소유자도 멤버 테이블에 통합 (기각)**: 쿼리는 한 테이블로 깔끔하나 전 프로젝트 owner행 백필 + `user_id` 중복/제거로 코드 광범위 변경. 글로서리상 "소유자도 멤버의 일종"은 개념으로 유지하되 저장은 분리.

### 접근 거부의 HTTP 의미론

- **역할 없음=404, 권한부족=403, 락/버전 충돌=409 (채택)**: 관계 없는 프로젝트는 기존처럼 존재를 숨겨 404. 보지만 못 하는 동작(뷰어의 쓰기 등)은 이미 존재를 아는 사용자라 403. 편집 충돌은 409로 분리.
- **전부 404 유지 (기각)**: 뷰어가 자기가 보는 프로젝트에 쓰기 시도 시 404는 어색하고 디버깅을 어렵게 한다.

## Consequences

- 새 테이블 `project_member`(`id`, `project_id` FK CASCADE, `user_id` FK CASCADE, `role` `editor|viewer`, `created_at`, UNIQUE(project_id,user_id))와 `project_edit_lock`(`project_id` PK/FK CASCADE, `locked_by` FK, `expires_at`), `project.version`(int, default 0) 컬럼, 그리고 Alembic 마이그레이션이 추가된다.
- 접근 제어가 "소유자 단일 스코프"(`get_by_id_and_user`)에서 "역할 해석"(`resolve_role` → owner/editor/viewer/none)으로 바뀐다. 모든 프로젝트·스냅샷·introspect 경로가 역할 기반 인가로 전환된다.
- 콘텐츠 쓰기(dbml/layout PATCH, DB 동기화, 스냅샷 복원)는 **편집 락 보유 + version 일치**를 요구한다. 비파괴 동작(스냅샷 생성, 조회, 내보내기)은 락이 불필요하다.
- 편집 락은 휘발성 런타임 상태로 **스냅샷에 포함되지 않으며** `project.updated_at`을 건드리지 않는다(별도 테이블).
- 초대는 **기존 가입 User만** 이메일로 즉시 부여(수락 단계·이메일 발송·보류 초대 없음). 이메일 가입 여부가 초대자에게 노출되는 약한 정보 누출은 내부 도구 성격상 수용한다. 미가입자 보류 초대는 향후 독립 추가.
- 소유자는 **1명·양도/공동소유 없음**(v1). 소유자는 "나가기"가 없고 삭제만 가능하다.
- 실시간 동시편집(B)으로의 이행은 이 ADR을 대체하는 별도 결정으로 다룬다 — 락/version은 그때까지의 정합성 장치다.
- ADR-0014(스냅샷)는 그대로지만, 복원은 이제 편집 락을 요구하고 스냅샷 이력 조회는 뷰어에게도 허용된다.
