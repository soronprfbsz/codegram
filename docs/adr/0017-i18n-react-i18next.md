# 사용자 노출 문자열은 react-i18next 기반 ko/en 단일 출처로 국제화한다

> **Status**: Accepted · **Tags**: `architecture`, `frontend` · **Date**: 2026-07-15 · **Supersedes**: — · **Related**: ADR-0007

Codegram UI의 모든 사용자 노출 문자열(라벨·버튼·플레이스홀더·`title`/`aria-label`·다이얼로그·토스트·검증/에러 메시지)을 **react-i18next**로 외부화하고, 번역은 `frontend/src/shared/i18n/locales/{ko,en}.json` **단일 출처**에 영역별로 둔다. 한국어가 제품의 기본 목소리이며 영어를 함께 지원한다. 컴포넌트는 `useTranslation()`의 `t('key')`로 **소비만** 하고 문자열을 직접 박지 않는다(프론트 규칙 F4의 근거).

이는 사후 기록이다 — 이미 `i18next@^26` + `react-i18next@^17`로 구현돼 있고(`shared/i18n/index.ts`), `useTranslation`을 쓰는 컴포넌트가 38개, ko/en 키가 30개 영역으로 완전 동기(ko-only/en-only 차집합 0)다. 왜 이렇게 정했는지를 남긴다.

## Considered Options

### 국제화 라이브러리

- **react-i18next (채택)**: React 생태계 표준. 보간(`t('k',{name})`)·복수형(`_one`/`_other`)·목록(`returnObjects`)·네임스페이스를 기본 제공하고, 런타임 언어 전환이 리렌더로 자연스럽게 반영된다. 번들은 `resources`에 ko/en JSON을 정적 import(코드 스플릿 불필요할 만큼 작음).
- **직접 만든 키→문자열 맵 (기각)**: 의존성 0이지만 보간·복수형·언어 전환 구독을 다 재구현해야 하고, 결국 react-i18next의 부분집합을 열등하게 다시 만드는 꼴.
- **하드코딩 + 나중에 국제화 (기각)**: 문자열이 JSX/속성에 흩어지면 언어 추가·문구 일괄 수정 때 누락·표류가 생긴다. 단일 출처 원칙(G1)에 반한다.

### 번역 저장 형태

- **영역별 섹션을 가진 ko/en JSON 단일 출처 (채택)**: `common`·`editor`·`groupSection`처럼 화면 영역별 최상위 섹션으로 묶어 키를 찾기 쉽게 한다. 새 문자열은 **ko/en 양쪽에 먼저 키를 추가한 뒤** 사용한다(둘 중 하나 누락 금지). 언어와 무관해야 하는 식별자(`data-testid`)는 번역에서 분리해 고정한다.
- **컴포넌트별 인라인/분산 번역 파일 (기각)**: 파일 수는 늘고, 언어 병렬성(ko↔en 대응) 검증이 어려워진다. 단일 파일 쌍이 diff·검수에 유리.

### 언어 감지·지속

- **localStorage(`erd-lang`) + ko 폴백 (채택)**: 테마 스토어와 같은 패턴으로 선택을 저장하고 동기적으로 초기 언어를 읽는다. `fallbackLng: 'ko'`.
- **브라우저 `navigator.language` 자동 감지 (기각)**: 제품 기본 목소리가 한국어라 명시적 저장·ko 우선이 더 예측 가능. 자동 감지는 추후 옵션으로 남김.

### 정책 강제

- **규칙 + 훅으로 강제 (채택)**: `.claude/rules/frontend.md` F4로 하드코딩 금지를 절대 규칙화하고, 리뷰/리팩터에서 하드코딩 문자열은 결함으로 회수한다.
- **관례로만 (기각)**: 강제가 없으면 하드코딩이 재유입돼 표류한다.
