# 프로젝트 스냅샷 히스토리는 전체 복사본을 인프로세스 스케줄러로 2단 보관한다

Project의 과거 상태를 되돌릴 수 있도록 **프로젝트 스냅샷**(`dbml_text` + `layout` 전체 복사)을 별도 테이블에 보관한다. 스냅샷은 세 종류(`kind`)다:

- **auto_fine**: 백엔드 주기 작업이 **30분마다** 변경된 프로젝트에 대해 생성, **3개월** 경과 시 삭제.
- **auto_coarse**: 백엔드 주기 작업이 **월 1회** 변경된 프로젝트에 대해 생성, **2년** 경과 시 삭제.
- **manual**: 사용자가 라벨을 붙여 명시적으로 생성, **영구 보관**(자동 삭제 제외), 프로젝트당 **최대 50개**, 사용자가 직접 삭제 가능.

결과 시간축: 최근 3개월은 30분 해상도, 3개월~2년은 월 해상도, 그 이전은 폐기(수동 스냅샷은 무관하게 영구). 원복은 선택한 스냅샷의 `dbml_text`+`layout`으로 현재 Project를 **덮어쓰기**(분기 없음)하며, 덮어쓰기 직전 현재 상태를 안전망 `auto_fine` 스냅샷으로 1개 먼저 찍는다.

## Considered Options

### 주기 작업 실행 방식

- **APScheduler 인프로세스 (채택)**: FastAPI lifespan에 스케줄러를 띄워 같은 프로세스에서 30분/월 작업을 돈다. 의존성 1개 추가로 끝, 별도 컨테이너·브로커 불필요. **전제: 백엔드 단일 인스턴스.** 다중 replica로 확장하면 각 인스턴스가 중복 실행하므로 DB advisory lock 또는 전용 워커 컨테이너로 이행해야 한다. 현재 배포는 단일 `codegram-backend` 컨테이너라 지금은 안전.
- **별도 cron/워커 컨테이너 (기각)**: 다중 web replica와도 안전하지만, 현재 단일 인스턴스 배포에 컨테이너·배포 구성을 하나 더 늘리는 것은 시기상조.
- **Celery beat (기각)**: 브로커(Redis)까지 필요 — 현재 스택에 없는 인프라를 통째로 들이는 것은 이 규모에 과함.

### 스냅샷 저장 표현

- **전체 복사 + 콘텐츠 해시 (채택)**: 스냅샷마다 `dbml_text` 전문 + `layout` JSON 통째로 저장하고, `content_hash`를 함께 보관해 자동 스냅샷 중복을 방지(수동은 중복 허용). 중복 비교는 **종류별**로 한다 — auto_fine은 마지막 auto_fine과, auto_coarse는 마지막 auto_coarse와 비교해 같으면 스킵. (단순히 종류 불문 직전과 비교하면, 변경 없는 달의 월 작업이 직전 fine과 같다고 스킵되고 그 fine이 3개월 뒤 삭제되어 **월 대표에 구멍**이 뚫린다.) 원복은 단순 복사 한 번이고 어느 스냅샷이든 독립 복원 가능. DBML은 텍스트라 용량이 작아 3개월 30분 단위라도 프로젝트당 수십 MB 수준.
- **차분(diff) 저장 (기각)**: 용량은 줄지만 원복 시 체인 재구성이 필요하고, 중간 스냅샷이 보관기한 경과로 삭제되면 체인이 끊긴다. 현 단계엔 과한 최적화 — 실제 용량 문제가 생기면 압축/차분 도입(되돌리기 쉬움).

### 자동 스냅샷의 변경 감지

- **콘텐츠 해시 비교 (채택)**: 편집했다 되돌려 내용이 동일하면 스킵. 정확하고 비용은 해시 1회.
- **`updated_at` 비교 (기각)**: 단순하지만 내용이 같아도 "변경됨"으로 잡혀 불필요한 중복이 생긴다.

### 월(coarse) 스냅샷 생성

- **독립 캡처 (채택)**: 30분 작업과 월 작업이 서로 독립. "어느 30분 스냅샷을 월 대표로 승격할지" 고민이 없고 prune이 단순. 월 경계의 fine·coarse 중복은 2년간 최대 24행으로 무시 가능.
- **승격/롤업 RRD 방식 (기각)**: 중복은 없지만 대표 선정 규칙과 prune 로직이 복잡.

## Consequences

- 새 테이블 `project_snapshot`(`id`, `project_id` FK ON DELETE CASCADE, `kind`, `label` nullable, `dbml_text`, `layout` JSONB, `content_hash`, `created_at`)과 Alembic 마이그레이션이 추가된다.
- 신규 엔드포인트: 스냅샷 목록 조회, 수동 스냅샷 생성, 수동 스냅샷 삭제, `POST /api/projects/:id/snapshots/:snapshotId/restore`(원자적 원복 — 안전망 스냅샷 생성 후 덮어쓰기).
- 원복 전 사용자는 선택한 스냅샷의 ERD를 **읽기 전용으로 미리보기**한다(현재 상태를 건드리지 않고 임시 렌더링하는 프론트 경로 필요).
- 스냅샷은 `dbml_text`+`layout`만 담는다 — `glyph`·`color`·`name`은 담지 않으며 원복해도 현재 값을 유지(글로서리 참조).
- prune 작업은 `kind`가 `auto_*`인 것만 대상으로 하고 `manual`은 건드리지 않는다.
- 앱 전반에 undo가 없는 것(ADR-0012)과 별개로, 스냅샷은 **거친 단위(30분~월)의 복구 수단**이지 세밀한 undo 대체가 아니다.
- APScheduler 인프로세스 전제는 **단일 인스턴스 배포에 묶인다** — 다중화 시 중복 실행 방지 장치를 반드시 추가해야 한다.
