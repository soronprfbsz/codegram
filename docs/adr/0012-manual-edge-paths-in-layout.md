# 관계선 수동 경로를 Layout에 절대좌표 꺾임점 목록으로 저장한다

관계선(엣지)의 수동 경로(Manual Path)는 `project.layout`(JSONB)의 `edges` 맵에 **엣지 키 → 절대좌표 꺾임점(waypoint) 배열**로 저장한다. 수동 경로를 가진 엣지는 자동 라우터(A* 직교 라우팅)를 건너뛰고 저장된 꺾임점으로만 경로를 그린다. 양 끝점은 저장하지 않고 컬럼 핸들에 동적으로 앵커된다. 경로는 수동·자동 모두 **항상 직교**(수평·수직 세그먼트만)이며, 렌더 시 비정렬 꺾임점 사이에 브리지 모서리를 자동 삽입해 불변식을 보장한다.

dbdiagram.io의 "Adjustable Relationship Lines"(2024-03)를 실측 검증한 결과를 따른다: 세그먼트 중간 핸들 드래그 → 수직 이동 + 인접 세그먼트 신축, 테이블 이동 시 꺾임점은 절대좌표로 보존되고 끝 세그먼트만 신축, 「Reset line」으로 자동 라우팅 복귀.

## Considered Options

- **절대좌표 꺾임점 + 자동 라우팅 제외 (채택)**: `StoredLayout.edges[edgeId].waypoints: {x,y}[]`. 테이블 이동 시 dbdiagram과 동일하게 동작(실측 `dbd-14`: 꺾임점 그 자리, 끝 세그먼트 신축). 끝점을 저장하지 않으므로 컬럼 순서 변경·테이블 이동에 끝점이 자동 추종. 렌더 분기가 단순("waypoints 있으면 그리기, 없으면 A*").
- **상대좌표 꺾임점 (기각)**: 어느 노드 기준인지(source? target? 중점?)가 자의적이고, 한쪽 테이블만 이동할 때 dbdiagram과 다른(직관에 반하는) 변형이 생긴다.
- **세그먼트 오프셋 모델 — A* 결과에 보정값만 저장 (기각)**: A*는 장애물(다른 테이블) 배치에 따라 꺾임 개수·위상이 바뀌므로 보정의 기준 경로 자체가 불안정. 테이블 하나만 옮겨도 보정이 무의미해진다.
- **연결면(좌/우)까지 저장 (기각 → 2026-06-12 부분 번복)**: dbdiagram 실측상 연결면은 항상 자동 결정이다. 저장하면 테이블이 반대편으로 이동했을 때 뒤로 꺾이는 경로를 강제하게 된다. **개정**: 사용자 기능 요청(엣지 선택 시 끝점별 좌/우 스왑 버튼)으로 `sourceSide`/`targetSide`를 **명시적 사용자 오버라이드일 때만** 저장한다(기본값 — source 우측/ target 좌측 — 은 저장하지 않음). 스왑은 해당 엣지의 수동 waypoints를 함께 비운다(앵커가 반대편으로 점프하면 기존 기하는 무의미). 기각 사유였던 "뒤로 꺾이는 경로"는 사용자가 스왑을 되돌리는 것으로 해소한다.

## Consequences

- `StoredLayout`에 선택적 `edges` 필드가 추가된다(`version: 1` 유지 — 필드 추가는 하위 호환). 엣지 키는 schemaToFlow의 이름 기반 id(`${ref.id}#${i}`)이므로 ADR-0004의 보존/소실 규칙이 그대로 적용된다: rename → 경로 소실 허용, DB 동기화 → 이름 기반 보존.
- Auto-arrange는 모든 수동 경로를 제거한다(위치 전면 재계산 후 기존 꺾임점은 무의미).
- 수동 경로 엣지는 노드 드래그 중 smoothstep 폴백이 필요 없다(꺾임점 고정 + 브리지 재계산은 A*와 달리 저비용).
- 사라진 관계의 고아 엔트리는 경로 커밋 시점에 현재 엣지 집합 기준으로 정리(prune)한다.
- undo는 제공하지 않는다(앱 전반에 undo 부재 — 일관성). 복구 수단은 「선 초기화」.
