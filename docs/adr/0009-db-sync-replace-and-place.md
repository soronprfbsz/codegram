# DB 동기화: 전체 교체 + 이름 기반 Layout 보존 + 신규 테이블 빈공간 배치

기존 Project의 ERD를 연결한 DB의 현재 구조로 갱신하는 "DB 동기화"(CONTEXT.md)는, introspect 결과 DBML로 **`dbml_text`를 전체 교체**하고, 사용자가 조정한 Layout(테이블 위치)은 **이름 기반으로 보존**한다(ADR-0004). DB에서 사라진 테이블은 다이어그램에서 제거되고, 남은 테이블은 위치를 유지하며, **신규 테이블은 기존(위치 보존) 테이블 bounding box 아래의 빈 영역에 compact dagre로 배치**하고 그 위치를 Layout에 명시 저장한다. Phase 1(가져오기)의 `/api/introspect`·`DbConnectDialog`를 재사용하되, 가져오기가 **새 Project**를 만드는 반면 동기화는 **현재 Project를 덮어쓴다**(덮어쓰기 확인 필요).

## Considered Options

- **DBML 갱신 방식**
  - **전체 교체 (채택)**: 단순·정직하고 "구조는 DB가 진실"과 일치. `@dbml/core` `ModelExporter`로 모델 머지 후 'dbml' 재직렬화도 가능하지만, 재직렬화가 모든 식별자를 따옴표로 감싸고 주석·사용자 포맷·줄 순서를 잃는 캐논화를 일으켜 사용자가 정리한 텍스트를 망친다 → 머지의 이점이 반감.
  - 구조 머지(주석 보존) (기각, 향후 가능): 살아남은 테이블의 노트/그룹/색을 이어받지만, 위 재직렬화 캐논화 + 리네임 감지 등 머지 로직·엣지케이스 비용이 크다.
- **신규 테이블 배치**
  - **기존 영역 아래 그리드/서브-dagre (채택)**: 기존 위치 불변·겹침 없음 보장·결정적. 신규 위치를 Layout에 저장해 reconcile이 그대로 honor하므로 매 렌더 dagre 재계산을 피한다.
  - 연관 테이블 근처 배치 (기각): 더 자연스럽지만 충돌 탐색이 복잡하고 밀집 시 실패.
  - 내부 빈 사각형 채우기(bin-packing) (기각): '기존 공백공간'에 가장 충실하나 구현 난도·불안정성이 높다.

## Consequences

- 전체 교체이므로 사용자가 DBML에 손으로 넣은 주석·수동 TableGroup·색은 동기화 시 사라진다(덮어쓰기 확인 다이얼로그로 경고). **Layout(위치)만 보존**된다.
- 이름 기반 reconcile(ADR-0004)의 귀결: DB에서의 테이블 리네임 = 다이어그램에서 **삭제 + 신규 추가**(옛 위치 잃고 빈 공간에 배치). 리네임 감지는 범위 밖.
- 동기화는 **단방향(DB→Project)·수동 트리거**다. Project의 변경을 DB로 되쓰지 않는다.
- **백엔드 변경 없음** — 기존 `/api/introspect`(ADR-0008)를 재사용한다. 신규 작업은 프론트엔드(빈공간 배치 유틸 + sync 오케스트레이션 + TopBar 진입점)에 한정된다.
- 신규 위치를 Layout에 저장한 뒤에는 기존 reconcile(ADR-0004)이 모든 테이블에 저장 위치를 갖게 되어, "신규 노드가 dagre 좌표로 떨어져 기존 노드와 겹치던" 문제를 동기화 경로에서 회피한다.
