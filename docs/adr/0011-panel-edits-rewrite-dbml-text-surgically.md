# 패널의 구조 편집은 DBML 텍스트를 국소 수술로 고친다 (모델 재직렬화가 아니라)

사이드 패널에서의 TableGroup 조작(생성·이동·색상·이름변경·삭제)은 구조 편집이므로 DBML 텍스트 변경으로 실행된다(ADR-0001, CONTEXT.md). 이때 파싱된 모델을 고쳐 전체 DBML을 재직렬화하는 대신, **현재 텍스트에서 해당 TableGroup 블록(또는 멤버 한 줄)만 찾아 국소적으로 삽입·삭제·치환**한다. DBML은 사용자가 직접 타이핑하는 소스라 주석·공백·선언 순서가 사용자 소유물이며, 재직렬화는 이를 전부 파괴하기 때문이다. `@dbml/core`에 exporter가 있음에도 쓰지 않는 것은 의도된 선택이다.

모든 수술은 **재파싱 가드**를 통과해야 적용된다: 고친 텍스트를 즉석에서 `parseDbml`로 검증하고, 실패하면 적용하지 않고 폐기한다(손상된 텍스트가 autosave로 저장되는 경로를 원천 차단). 같은 이유로 현재 텍스트가 Invalid인 동안(패널은 lastValidSchema를 표시 — 화면과 텍스트가 어긋난 상태) 패널의 구조 편집 버튼은 비활성화한다 — 읽기 전용 동작(선택·접기)만 허용.

## Considered Options

- **국소 텍스트 수술 (채택)**: 주석·포맷·순서 보존. 블록 탐색/치환 로직과 엣지 케이스(따옴표 이름 `"인증 그룹"`, 스키마 한정 멤버 `myschema.users`, 그룹 내 `Note:` 줄 보존)를 직접 다뤄야 하는 비용이 있다.
- 모델 수정 후 전체 재직렬화 (기각): 구현은 단순하나 사용자가 손으로 쓴 주석·수동 포맷·선언 순서가 소멸한다. "DBML 텍스트가 사용자 소유의 소스"라는 ADR-0001 전제와 충돌.

## Consequences

- 수술 함수들은 순수 텍스트 연산(`entities/dbml`)으로 두고 단위 테스트로 엣지 케이스를 잡는다. 파서 검증 규칙: TableGroup 이름은 스키마 내 유일(중복 거부), 테이블은 최대 1그룹(파서 강제), 빈 그룹 허용.
- UI는 변경 후 텍스트를 기존 `setDbmlText` 경로로 흘려보내므로 에디터·캔버스·패널 반영과 autosave는 추가 배선 없이 동작한다.
