# Export는 데이터 의존성으로 분리한다 — Diagram은 에디터, Table Doc·SQL은 프로젝트 컨텍스트 메뉴

기존 에디터 상단바의 통합 "Actions" 드롭다운을 해체하면서, 그 안의 Export 항목을 한 곳에 모으지 않고 **데이터 의존성에 따라 두 곳으로 나눈다**. Diagram(PNG/SVG/PDF)은 마운트된 React Flow 캔버스를 캡처해야 하므로 **현재 에디터에 열린 프로젝트에서만** 가능하고 그 캡처 핸들이 `EditorPage`에 있으므로 에디터 상단바에 남긴다. Table Doc(HTML/Excel/PDF)·SQL은 `dbml_text` 파싱만으로 생성되어 열지 않은 프로젝트도 대상이 되므로, 모든 프로젝트 행에 붙는 **사이드바 "⋯" 컨텍스트 메뉴**로 옮긴다(목록 응답 `list[ProjectRead]`가 이미 `dbml_text`를 포함).

## Considered Options

- **데이터 의존성으로 분리 (채택)**: Diagram→에디터, Table Doc·SQL→사이드바 컨텍스트 메뉴. 각 산출물이 필요로 하는 데이터가 있는 곳에 둔다. 가장 단순하고 기존 아키텍처(캔버스 캡처는 `EditorPage` 소유)와 일치.
- **전체 Export를 컨텍스트 메뉴에, Diagram은 활성 프로젝트만 (기각)**: 에디터의 캡처 핸들을 전역 위젯인 사이드바로 넘기는 전역 스토어/컨텍스트가 필요하고, 프로젝트마다 Diagram 활성/비활성이 달라져 메뉴가 일관되지 않음.
- **전체 Export를 컨텍스트 메뉴에, 비활성 프로젝트는 열고 나서 캡처 (기각)**: navigate→캔버스 마운트 대기→fitView→캡처의 다단계·타이밍 의존이 취약.

## Consequences

- "Export"가 두 위치(에디터 상단바 · 사이드바 컨텍스트 메뉴)에 흩어진다. 이유는 산출물의 데이터 의존성이며, 이 ADR이 그 근거다.
- 사이드바 컨텍스트 메뉴는 해당 행 프로젝트의 `dbml_text`를 그 자리에서 파싱한다. 현재 편집 중인 프로젝트는 자동저장 디바운스 창(≈1초) 동안 목록 캐시가 직전 편집보다 뒤처질 수 있으나(별도 flush 없이) 수용한다.
- 스키마가 비면(테이블 0개) export 항목은 비활성(`exportDisabled`와 동일 게이트).
