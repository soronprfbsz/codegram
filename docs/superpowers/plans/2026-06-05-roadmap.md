# Codegram — Implementation Roadmap

> Plan 0 has a detailed companion document: [`docs/superpowers/plans/2026-06-05-00-foundation-scaffolding.md`](./2026-06-05-00-foundation-scaffolding.md).

This roadmap sequences the seven plans that take Codegram from an empty repository to a complete DBML-driven ERD editor with exports and SQL interop. Each plan builds strictly on its predecessors. Plans are vertical slices that ship working, tested software; they are not horizontal "all backend, then all frontend" phases.

---

## Cross-cutting conventions

These apply to **every** plan and are not repeated below.

- **Source-of-truth boundary (ADR-0001).** DBML text is the single source of truth for ERD structure. There is NO visual-to-DBML structural round-trip: dragging/editing the canvas never rewrites table/column/relationship definitions. Only `layout` (positions, waypoints) flows back from the canvas. Any structural change happens by editing DBML text.
- **DBML parsing locus (ADR-0002).** DBML is parsed in the **frontend** using official `@dbml/core`. The backend never parses DBML; it stores `dbml_text` as an opaque `TEXT` column.
- **Layout reconciliation (ADR-0004).** On each parse, layout is merged with parse output using **NAME-based keys** (table/column/enum names). Unpositioned nodes get auto-layout (dagre); positioned nodes keep their stored coordinates. A rename loses the old position (treated as a new node). Reconciliation is a pure merge run on every successful parse.
- **Autosave.** Persistence is debounced autosave of `dbml_text` + `layout`. No manual save button, no version history.
- **FSD import rule (frontend).** Layers import strictly downward only: `app > pages > widgets > features > entities > shared`. No same-layer cross-imports; `shared` imports nothing upward. Enforced by convention (and ESLint where configured).
- **Backend layering.** Single-direction dependency: `router (FastAPI routes) -> service (business logic) -> repository (data access)`. Pydantic v2 schemas are DTOs at the router boundary; SQLAlchemy models live only in repository/models. Routers never touch the DB session directly for business logic.
- **TDD (strict).** Write the failing test first, then implement. Backend: pytest + pytest-asyncio. Frontend unit: Vitest + Testing Library. Core flows: Playwright E2E. Every plan's ship criteria are expressed as passing tests.
- **Conventional commits.** All commits follow the Conventional Commits spec (`feat:`, `fix:`, `test:`, `chore:`, `docs:`, `refactor:`), scoped where useful (e.g. `feat(editor): ...`).

## Plan 0 — Foundation & Scaffolding

**Goal.** Stand up the monorepo skeleton (backend, frontend, postgres) with a green health check and smoke tests, so all later plans have a working, dockerized base to build on.

**Key files / areas touched.**
- Root: `git init`, `.gitignore`, `.env.example`, `README.md`, `docker-compose.yml`.
- Backend scaffold: `backend/pyproject.toml`, `backend/Dockerfile`, `backend/alembic.ini`, `backend/alembic/env.py`, `app/main.py` (app factory + CORS), `app/core/config.py`, `app/db/base.py`, `app/db/session.py`, `app/api/router.py`, `app/api/routes/health.py`, `app/schemas/health.py`, placeholder packages (`app/services`, `app/repositories`, `app/models`), `tests/conftest.py`, `tests/test_health.py`.
- Frontend scaffold: `frontend/package.json`, `frontend/Dockerfile`, `index.html`, `vite.config.ts`, `vitest.config.ts`, `tsconfig*.json`, `components.json` (shadcn), `playwright.config.ts`, FSD skeleton `src/app/`, `src/pages/home/`, `.gitkeep` for `widgets/features/entities`, `src/shared/api/client.ts`, `src/shared/config/env.ts`, `src/shared/store/ui.ts`, `src/shared/ui/`, `e2e/home.spec.ts`.

**External libs introduced.**
- Backend: FastAPI, uvicorn[standard], SQLAlchemy 2.0, asyncpg, Alembic, Pydantic v2, pydantic-settings, pytest, pytest-asyncio, httpx, anyio.
- Frontend: React 19, Vite 8, TypeScript 6, TanStack Query, Zustand, React Router 7, shadcn (preset `b1FlFygMM`), Tailwind, Vitest, Testing Library, Playwright.
- Infra: Docker Compose (postgres:17-alpine, python:3.13-slim, node:22-alpine).

**Dependencies.** None (first plan).

**Ship criteria.** `docker compose up` brings up postgres + backend + frontend; `GET /api/health` returns OK after a DB `SELECT 1`; the Home page renders. Tested by: `pytest` (health endpoint hits test DB session), Vitest (Home page smoke render), Playwright (`home.spec.ts` loads `/`).

## Plan 1 — Authentication

**Goal.** Provide multi-user signup/login with per-user isolation using fastapi-users (email+password, JWT in an httpOnly cookie).

**Key files / areas touched.**
- Backend: `app/models/user.py` (fastapi-users SQLAlchemy user table), `app/core/users.py` (UserManager, auth backend, cookie transport, JWT strategy), `app/api/routes/auth.py` (register/login/logout/me routers), `app/core/config.py` (SECRET_KEY, token TTL), first real Alembic migration in `alembic/versions/`, `app/schemas/user.py` (UserRead/UserCreate). Auth dependency (`current_active_user`) for later route protection.
- Frontend: `src/entities/session/` (current-user query + types), `src/features/auth/` (login form, register form, logout), `src/pages/login/`, `src/pages/register/`, route guards in `src/app/providers/router.tsx`, API client cookie handling in `src/shared/api/client.ts`.

**External libs introduced.** `fastapi-users[sqlalchemy]` (backend). No major new frontend libs (uses existing Query/Router/shadcn forms).

**Dependencies.** Plan 0 (DB session, app factory, FSD/router skeleton, test fixtures).

**Ship criteria.** A user can register, log in (receives httpOnly JWT cookie), see their own session, and log out; protected endpoints reject unauthenticated requests. Tested by: pytest (register/login/me/logout flows, 401 on protected routes), Vitest (auth form components), Playwright (register -> login -> see authenticated home -> logout).

## Plan 2 — Project CRUD + Autosave

**Goal.** Let an authenticated user create, list, open, rename, and delete projects, each holding `dbml_text` + `layout`, with debounced autosave persisting changes.

**Key files / areas touched.**
- Backend: `app/models/project.py` (Project: id, user_id FK, name, `dbml_text` TEXT, `layout` JSONB, timestamps), `app/repositories/project.py`, `app/services/project.py` (ownership enforcement, per-user isolation), `app/schemas/project.py` (ProjectCreate/Read/Update + autosave payload), `app/api/routes/projects.py` (CRUD + PATCH for autosave), Alembic migration for the projects table.
- Frontend: `src/entities/project/` (types, TanStack Query hooks: list/get/create/update/delete), `src/features/project-list/`, `src/features/project-autosave/` (debounced mutation), `src/pages/home/` (project list/dashboard), `src/pages/editor/` (shell that loads a project), `src/shared/hooks/useDebounce.ts`.

**External libs introduced.** None new (reuses TanStack Query, SQLAlchemy JSONB, existing stack).

**Dependencies.** Plan 1 (user identity + auth dependency for ownership) and Plan 0 (DB/migrations, FSD layers).

**Ship criteria.** A logged-in user creates a project, opens its editor shell, and edits to `dbml_text`/`layout` are autosaved (debounced) and survive reload; users cannot access each other's projects. Tested by: pytest (CRUD + ownership/403 isolation, autosave PATCH), Vitest (project hooks + debounced autosave), Playwright (create project -> edit -> reload -> persisted).

## Plan 3 — DBML Editor + Parsing + ERD Render

**Goal.** Add the CodeMirror DBML editor with debounced live parsing (`@dbml/core`) and render the parsed schema as a React Flow ERD (tables, columns, relationships, enums, table groups, sticky notes).

**Key files / areas touched.**
- Frontend (the heart of this plan): `src/features/dbml-editor/` (CodeMirror 6 editor, debounced parse, inline parse-error markers), `src/entities/dbml/` (parse adapter over `@dbml/core`, normalized schema model, error types), `src/widgets/erd-canvas/` (React Flow v12 setup), `src/entities/erd/ui/` custom nodes (TableNode with per-column handles, EnumNode, StickyNoteNode) and custom crow-foot edges, table-group colored regions, `src/features/erd-preview/` wiring editor -> parse -> canvas. Editor page composition in `src/pages/editor/`.
- Backend: none structurally (DBML stays opaque TEXT per ADR-0001/0002); only consumes existing autosave.

**External libs introduced.** `@dbml/core`, `@xyflow/react` (React Flow v12), CodeMirror 6 (`@codemirror/*`).

**Dependencies.** Plan 2 (project holds `dbml_text`, autosave path) and Plan 0/1.

**Ship criteria.** Typing DBML in the editor updates a rendered ERD live (debounced); valid schemas render tables with per-column handles, crow-foot relationship edges, enums, and colored table groups; parse errors show inline markers without crashing the canvas. Tested by: Vitest (parse adapter for valid/invalid DBML, node/edge mapping, error markers), Playwright (type DBML -> see corresponding tables and a relationship edge render).

## Plan 4 — Manual Layout + Reconciliation

**Goal.** Persist manual layout (drag tables, edit edge waypoints with auto-reset) and reconcile stored layout with parse output on every parse via name-based merge plus dagre auto-layout for unpositioned nodes (ADR-0004).

**Key files / areas touched.**
- Frontend: `src/features/layout-persistence/` (capture node positions + edge waypoints into `layout`, feed autosave from Plan 2), `src/entities/layout/model/` (reconciliation merge: name-keyed merge of parsed nodes with stored positions, drop-on-rename, waypoint auto-reset), `src/features/auto-layout/lib/` (dagre layered layout for unpositioned nodes), wiring into `src/widgets/erd-canvas/` (drag handlers, controlled node positions) and the parse pipeline from Plan 3.
- Backend: none new (`layout` JSONB already exists; autosave PATCH from Plan 2 carries it).

**External libs introduced.** `dagre` (or `@dagrejs/dagre`) for auto-layout.

**Dependencies.** Plan 3 (parsed schema + canvas nodes/edges) and Plan 2 (autosave of `layout`).

**Ship criteria.** Dragging a table persists its position across reloads; newly added (unpositioned) tables get a sensible dagre layout; renaming a table drops its old position (reconciliation by name); edge waypoints can be edited and auto-reset. Tested by: Vitest (pure reconciliation merge: positioned-kept, unpositioned-laid-out, rename-loses-position; waypoint reset logic), Playwright (drag table -> reload -> position retained; add table via DBML -> auto-positioned).

## Plan 5 — Export

**Goal.** Deliver client-side exports: diagram as PNG/SVG (and PDF), plus a table-definition document as an in-app HTML view, Excel, and PDF.

**Key files / areas touched.**
- Frontend (all client-side per ADR-0005): `src/features/export-diagram/` (PNG/SVG via html-to-image over the React Flow canvas; PDF via jsPDF), `src/features/export-table-doc/` (derive table-definition rows from the normalized schema; in-app HTML view; Excel via SheetJS; PDF via jsPDF), `src/widgets/table-doc-view/` (HTML definition view), export UI/dialogs and triggers in `src/pages/editor/` toolbar.
- Backend: none (exports never touch the server).

**External libs introduced.** `html-to-image`, `jspdf`, `xlsx` (SheetJS).

**Dependencies.** Plan 3 (rendered canvas + normalized schema for the doc) and Plan 4 (final laid-out canvas to capture).

**Ship criteria.** From the editor a user exports the diagram to PNG/SVG/PDF and the table-definition document to HTML view/Excel/PDF, entirely in the browser. Tested by: Vitest (table-doc derivation from schema; export builders invoked with correct data; mocked exporters), Playwright (trigger diagram export and table-doc export -> download/asset produced).

## Plan 6 — SQL <-> DBML Import/Export

**Goal.** Add SQL import (SQL -> DBML) and SQL export (DBML -> SQL) using `@dbml/core`'s importer/exporter, populating or generating from the project's `dbml_text`.

**Key files / areas touched.**
- Frontend: `src/features/sql-import/` (paste/upload SQL -> `@dbml/core` import -> set `dbml_text`, then the existing parse/render/reconcile pipeline takes over), `src/features/sql-export/` (current `dbml_text` -> `@dbml/core` export -> downloadable SQL, dialect selection), import/export UI in `src/pages/editor/` toolbar, error surfacing reusing the parse-error pattern from Plan 3.
- Backend: none (conversion is client-side via `@dbml/core`, consistent with ADR-0001/0002).

**External libs introduced.** None new (reuses `@dbml/core` from Plan 3; possibly its importer submodules).

**Dependencies.** Plan 3 (parse/render pipeline and `@dbml/core` integration), Plan 2 (writing `dbml_text` via autosave), Plan 5 patterns for file download.

**Ship criteria.** A user imports a SQL schema and sees it become editable DBML that renders as an ERD, and exports the current DBML to SQL for a chosen dialect. Tested by: Vitest (SQL->DBML import adapter and DBML->SQL export adapter for representative schemas, including malformed-SQL error handling), Playwright (import SQL -> ERD renders; export -> SQL file produced).

## Sequencing summary

```
Plan 0 (foundation)
  -> Plan 1 (auth)
       -> Plan 2 (project CRUD + autosave)
            -> Plan 3 (editor + parse + render)
                 -> Plan 4 (layout + reconciliation)
                      -> Plan 5 (export)
                      -> Plan 6 (SQL <-> DBML)   # depends on Plan 3; benefits from Plan 5 download patterns
```

Plans 5 and 6 both depend on Plan 3's parse/render core and Plan 4's stable canvas; they can be developed in either order after Plan 4.
