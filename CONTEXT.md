# Codegram

DBML 텍스트로부터 ERD(개체-관계 다이어그램)를 생성·편집·출력하는 웹 서비스. DBML 텍스트가 구조의 단일 진실 공급원이며, 사용자가 조정한 시각적 배치는 별도로 보관된다.

## Language

**DBML** (Database Markup Language):
테이블과 관계를 선언하는 텍스트 DSL. 이 서비스에서 **구조(테이블·컬럼·관계)의 유일한 진실 공급원**이다. 구조 변경은 오직 DBML 편집으로만 이루어진다.
_Avoid_: 스키마 텍스트, 소스

**ERD**:
DBML을 렌더링한 시각적 다이어그램. 그 자체로 진실이 아니라 DBML의 표현물이다.
_Avoid_: 다이어그램, 도식 (단독 사용 시 모호함)

**Layout** (배치):
사용자가 수동으로 조정한 시각적 좌표 정보 — 캔버스 객체(테이블 노드·Enum 노드·스티키 노트)의 위치, 관계선(엣지)의 수동 경로. DBML과 **분리되어** 저장되며 진실 공급원이 아니다. DBML이 바뀌어도 가능한 한 보존된다(이름 기반 키, ADR-0004).
경계 원칙: **DBML이 표현할 수 없는 것만 Layout에 둔다**(x/y 좌표, 엣지 경로). DBML이 네이티브로 표현 가능한 것(구조·그룹·색상)은 Layout이 아니라 DBML에 둔다.
직접 배치(드래그·좌표 편집)의 대상은 테이블·Enum 노드·스티키 노트·관계선 경로뿐이다 — **그룹 박스의 위치·크기는 멤버 테이블로부터 파생**되므로 그 자체를 좌표 편집하거나 임의 위치로 둘 수 없다. 다만 그룹 **라벨**을 끌면 그룹에 속한 멤버 테이블 전체가 함께 평행이동하는 편의 동작은 허용되며(=멤버 일괄 이동), 이때도 박스의 위치·크기는 이동된 멤버들로부터 다시 파생된다. 그룹 영역의 라벨 외 부분은 클릭·드래그를 가로채지 않고 아래의 관계선·핸들로 통과시킨다. 관계선의 연결면(테이블의 어느 변에 붙는지)은 기본 자동이되 사용자가 끝점별 좌/우 스왑으로 바꿀 수 있다. 사용자에게 보이는 좌표는 항상 **절대 캔버스 좌표**다(그룹 멤버 여부와 무관).
_Avoid_: 포지션, 좌표 (Layout이 더 넓은 개념)

**수동 경로** (Manual Path):
사용자가 직접 조정한 관계선(엣지)의 경로. 기본은 자동 계산 경로이며, 사용자가 선을 조정하는 순간 그 엣지는 수동 경로를 갖고 자동 계산 대상에서 빠진다. "조정"은 캔버스에서의 드래그든 Info 패널에서의 꺾임점 좌표 수정이든 동일하게 취급된다. 경로는 **꺾임점**(waypoint) 목록으로 표현되며, 양 끝점은 컬럼에 앵커되어 꺾임점이 아니고 편집할 수 없다. 경로는 수동·자동을 불문하고 **항상 직교**(수평·수직 세그먼트만)이며 대각선 세그먼트는 존재하지 않는다. 수동 경로는 구조가 아니므로 DBML이 아니라 Layout에 저장되고(엣지 키 기반, ADR-0004), **선 초기화(Reset line)**로 제거하면 해당 엣지는 자동 계산으로 되돌아간다.
생존 규칙: 연결된 테이블이 이동해도 waypoint는 절대좌표로 **유지**되고 끝점만 컬럼을 따라간다(끝 세그먼트 신축) — 진행 방향이 뒤집혀 모양이 흉해져도 말없이 버리지 않는다(정리는 사용자의 선 초기화로만). 단, **전체 자동 정렬(Auto-arrange)은 모든 수동 경로를 제거**하고, **그룹별 자동 정렬은 그 정렬로 이동한 멤버에 닿는 엣지(한쪽 끝점이라도 해당 멤버인 경우)의 수동 경로만 제거**한다(국소 정렬이므로 영향 범위도 국소). 엣지 키가 바뀌는 변경(테이블·컬럼 rename)은 위치와 마찬가지로 수동 경로를 잃으며(ADR-0004), DB 동기화는 남아 있는 관계의 수동 경로를 이름 기반으로 보존한다(ADR-0009).
_Avoid_: 커스텀 라우팅(구현 용어), 선 고정 (초기화로 되돌릴 수 있으므로 "고정"이 아니다)

**TableGroup** (테이블그룹):
연관 테이블을 한 영역으로 묶는 DBML 네이티브 개념. `TableGroup name [color: #hex] { ... }`로 영역 배경색과 Note를 가진다. 구조·색상이 모두 DBML 안에 있으므로 진실 공급원은 DBML이며, 앱은 파싱 결과로 색상 영역을 렌더링한다. (테이블별 헤더색은 `Table x [headercolor: #hex]`.) 한 테이블은 **최대 하나의 TableGroup**에만 속한다(파서가 강제) — 따라서 "이동"은 언제나 "이전 그룹에서 제거 + 새 그룹에 추가"이고 그 외의 상태는 없다. 패널 리스트에서의 그룹 접힘/펼침은 구조도 배치도 아닌 휘발성 UI 상태로, DBML에도 Layout에도 저장하지 않는다.
_Avoid_: 색상 그룹, 영역 (TableGroup이 정식 명칭)

**Ungrouped** (미분류):
어느 TableGroup에도 속하지 않은 테이블들을 모아 보여주는 **파생 버킷**. DBML에 존재하는 엔티티가 아니라 화면 표시용으로만 계산되므로 이름 변경·색상·삭제가 없고, "그룹에서 빼기"와 "Ungrouped로 이동"은 같은 동작(그룹 블록에서 해당 테이블 제거)이다.
_Avoid_: 기본 그룹, 무소속 그룹 (그룹이 아니다)

**User** (사용자):
인증된 계정. 여러 Project를 소유하고, 다른 User가 **공유**한 Project에 **멤버**로 참여할 수 있다. 자신이 소유하거나 멤버로 속한 Project만 접근하며, 그 외에는 격리된다.

**Project** (프로젝트, 워크스페이스 단위):
정확히 한 명의 **소유자(Owner)**가 가지는 작업 단위. **정확히 하나의 DBML 문서 + 그 Layout + 테이블그룹 스타일**로 구성되고, **하나의 ERD**로 렌더링된다. 소유자는 다른 User를 멤버로 초대해 함께 접근하게 할 수 있다(역할별 권한). 별도의 "ERD/Diagram" 저장 엔티티는 두지 않는다.
_Avoid_: DBML `Project` 블록과 혼동 금지 (아래 참조)

**공유 (Share)**:
소유자가 다른 User에게 특정 Project의 접근 권한을 **역할**과 함께 부여하는 행위. 항상 **Project 단위**이며, 한 User↔Project 쌍에는 역할이 **하나만** 정해진다.
_Avoid_: 협업(범위가 더 넓은 말), 퍼블리시·게시(공개 노출이 아니라 지정 User 초대다)

**프로젝트 멤버 (Project Member)**:
한 User가 한 Project에 대해 갖는, 역할이 부여된 접근 관계. 소유자도 멤버의 한 종류(소유자 역할)로 본다. 한 User는 한 Project에 대해 **하나의 역할만** 갖는다.
_Avoid_: 협업자, 게스트, 참가자 (멤버가 정식 명칭)

**역할 (Role)**:
멤버가 한 Project에 대해 갖는 권한 등급(**Project 스코프**). 정확히 3종이며 Project당 한 명에게 하나만 부여된다.
- **소유자 (Owner)**: 생성자. Project당 **정확히 1명**(양도·공동소유 없음). 구조/배치 편집 + 멤버 관리(초대·제거·역할 변경) + Project 삭제까지 전권.
- **편집자 (Editor)**: 구조/배치 편집 가능. 멤버 관리·Project 삭제는 불가.
- **뷰어 (Viewer)**: 조회·내보내기만 가능한 읽기 전용. 어떤 영속 변경도 못 한다.
_Avoid_(이 Project 역할 문맥에 한정): 권한 레벨·퍼미션(역할이 정식 명칭), "관리자"(소유자와 구분되는 별도 등급이 아니다). ↔ 아래 **시스템 역할**은 이와 별개의 전역 개념이다.

**시스템 역할 (System Role)**:
계정(User)이 앱 전역에서 갖는 등급. **Project 역할과 직교**하며(ADR-0016), 계정 관리·비밀번호 초기화 권한을 가른다. 계정당 정확히 **하나**. 2종:
- **관리자 (admin)**: 계정 목록 조회 + 계정 관리(역할 변경·비밀번호 초기화)·권한 매트릭스 편집.
- **일반 (user)**: 계정 목록 조회만. 신규 가입 계정의 기본값.
_Avoid_: (Project) 소유자/편집자/뷰어와 혼동 — 시스템 역할은 Project 접근과 무관하다.

**권한 (Permission)**:
시스템 역할이 갖는 세분 능력(전역). 고정 카탈로그 2종: `user:read`(계정 목록 조회), `user:manage`(계정 관리·권한 편집). 역할↔권한은 편집 가능(자기잠금 가드: admin은 `user:manage`를 잃지 않음). ADR-0016.

**테이블 정의서** (Table Definition Document, 데이터 사전):
파싱된 모델에서 도출하는 문서형 산출물. 테이블별로 컬럼(이름·타입·PK·FK·NN·UQ·기본값·설명)·관계·인덱스·그룹을 정리한 데이터 사전. 인앱 HTML 표 뷰로 보고 Excel(.xlsx)·PDF로 내보낸다. ERD(시각 다이어그램)와 구분되는 별개의 출력물.
_Avoid_: 스키마 문서, 명세서 (정의서가 정식 명칭)

**DB 가져오기** (Database Import / Schema Introspection):
실행 중인 외부 데이터베이스(PostgreSQL·MariaDB·ClickHouse)에 접속해 스키마를 읽어(introspection) 그 구조를 새 Project의 DBML로 옮기는 동작. 접속 정보(호스트·유저·비밀번호)는 **1회용**으로만 쓰이고 저장되지 않는다. 백엔드가 외부 DB에 접속해 스키마를 읽는 방식은 dialect별로 다르다 — PostgreSQL·MariaDB는 reflection으로 **DDL**을 만들고(프론트 `@dbml/core`가 DDL→DBML 변환), ClickHouse는 `system.tables`/`system.columns`를 읽어 **구조화된 테이블·컬럼 목록**을 반환하고 프론트가 그것으로 DBML을 직접 만든다(ADR-0021). 어느 경우든 **DBML 의미는 여전히 프론트에서만 다룬다**(ADR-0002 유지). ClickHouse는 외래키가 없어 **관계 없는 테이블·컬럼만** 옮긴다. 한 번의 가져오기는 한 schema(PG는 기본 `public`, MariaDB·ClickHouse는 접속한 database)를 대상으로 하고 결과는 새 Project 하나가 된다.
_Avoid_: DB 연동 (가져오기는 새 Project를 만드는 1회성 동작 — 기존 Project를 갱신하는 것은 아래 **DB 동기화**다)

**DB 동기화** (Sync):
**기존 Project**의 구조를 연결한 DB의 현재 상태로 갱신하는 동작. DBML은 introspect 결과로 **전체 교체**되며(구조의 진실 공급원은 DB, 단방향), 사용자가 조정한 Layout(테이블 위치)은 **이름 기반으로 보존**된다(ADR-0004): DB에서 사라진 테이블은 다이어그램에서 제거되고, 남은 테이블은 위치를 유지하며, **신규 테이블은 기존 배치를 건드리지 않고 다이어그램의 빈 공간에 배치**된다. 가져오기와 같은 `/api/introspect` 경로·접속 다이얼로그를 재사용하지만, 가져오기가 **새 Project를 만드는** 반면 동기화는 **현재 Project를 덮어쓴다**. 전체 교체이므로 사용자가 DBML에 손으로 넣은 주석·수동 TableGroup·색은 보존되지 않는다.
_Avoid_: 양방향 동기화, 라이브 동기화 (DB→Project 단방향·수동 트리거이며, Project의 변경을 DB로 되쓰지 않는다)

**프로젝트 스냅샷** (Project Snapshot):
한 시점의 Project 상태를 되돌릴 수 있도록 보관한 사본. 담는 것은 **`dbml_text` + `layout`뿐**이다 — 완전 원복에 필요한 최소이자 충분한 집합. `glyph`·`color`·`name`은 "그 시점의 ERD 모습"과 무관한 메타데이터이므로 스냅샷에 담지 않으며, 원복해도 현재 값을 유지한다. 스냅샷은 진실 공급원이 아니라 과거 상태의 읽기 전용 기록이다. (요청자가 "ERD history"라 불렀으나, ERD는 DBML의 렌더링일 뿐이므로 되돌림의 대상은 ERD가 아니라 그것을 만들어내는 재료인 Project다.)
_Avoid_: ERD 히스토리, 버전 (스냅샷이 정식 명칭 — "버전"은 Layout `version: 1` 필드와 혼동)

## Flagged ambiguities

- **"편집"의 두 종류**: _구조 편집_(테이블/컬럼/관계/TableGroup 추가·삭제·변경)은 DBML 텍스트 변경으로만 이루어진다. 직접 타이핑 외에, UI가 사용자를 대신해 DBML 텍스트를 고치는 것(예: 패널에서의 TableGroup 조작)도 구조 편집이다 — 이 경우에도 진실 공급원은 여전히 DBML 텍스트다. _배치 편집_(위치·선 경로 조정)은 다이어그램에서 직접 가능하다. 두 가지를 혼용하지 말 것.
- **Project (앱) vs DBML `Project` 블록**: 앱의 **Project**는 사용자의 워크스페이스 단위(소유·저장 대상)다. DBML의 `Project { database_type ... Note ... }`는 한 DBML 문서 안의 **문서 수준 메타데이터** 블록이다. 둘은 다른 개념이며, 앱 Project 하나가 DBML 문서 하나(그 안에 DBML `Project` 블록을 0~1개 포함)에 대응한다.

## Example dialogue

> **개발자**: 사용자가 다이어그램에서 테이블을 드래그하면 DBML이 바뀌나요?
> **도메인 전문가**: 아니요. 드래그는 **Layout**만 바꿉니다. DBML은 그대로예요. 위치 좌표는 따로 저장됩니다.
> **개발자**: 그럼 다이어그램에서 컬럼을 추가하고 싶으면요?
> **도메인 전문가**: 그건 _구조 편집_이라 다이어그램에서는 못 합니다. **DBML** 텍스트를 고쳐야 해요. 고치면 ERD가 다시 그려지고, 그때 기존 **Layout**은 최대한 유지됩니다.
